个人博客很容易被当成一个“写完就发布”的静态页面，但如果希望它长期稳定运行，最好把它当作一个小型工程项目来维护。

这意味着每一次改动都要回答三个问题：

- 读者是否更容易找到有价值的内容？
- 站点是否更稳定、更容易部署？
- 后续写作和维护是否更省心？

## 背景

这个博客同时面向 GitHub Pages 和 Vercel。两个平台的路径模型不同：GitHub Pages 部署在 `/ryuho/` 子路径，Vercel 部署在站点根路径。如果页面里写死链接，很容易出现一个平台正常、另一个平台失效的问题。

因此，第一步不是做复杂功能，而是把上线流程变成可验证的工程流程。

## 方案

我把优化拆成三层：

1. 站点定位：让首页一眼说明这是技术博客，而不是泛泛的个人主页。
2. 发布质量：构建后自动检查 SEO、内部链接、静态资源和关键 CTA。
3. 内容质量：避免默认示例、空模板和没有上下文的文章被发布出去。

这些检查不依赖外部服务，只扫描生成后的 `public` 目录。好处是速度快、失败原因明确，也能同时覆盖 GitHub Pages 和 Vercel 两种构建结果。

## 实现

核心思路是把 `npm run check` 当成本地和 CI 的统一入口：

```bash
npm run check
npm run check:vercel
```

前者使用 GitHub Pages 的子路径配置，后者使用 Vercel 的根路径配置。两者都会在生成静态站点后运行审计脚本。

审计脚本重点检查：

- 首页是否具备核心 SEO 标签
- `robots.txt`、`sitemap.xml`、`atom.xml` 是否生成
- 内部链接和静态资源是否都能解析到真实文件
- manifest 图标声明是否和真实文件类型一致
- 首页“阅读最新文章”是否真的指向文章详情
- 生成后的 HTML 是否还残留示例内容或空日记模板

## 验证

静态站点最怕“构建成功但线上体验坏了”。所以每次提交前，至少要确认两个命令都通过：

```bash
npm run check
npm run check:vercel
```

如果其中一个失败，就说明当前改动还没有达到可发布状态。这个规则比口头提醒可靠，因为它会在构建产物层面直接给出失败项。

## 复盘

个人博客的价值不只在文章数量，也在长期可维护性。

一个优秀的 IT 个人博客应该像一个小产品：定位清楚、路径顺畅、内容可信、发布稳定。等这些基础打牢之后，再去做评论、搜索、统计、项目页等增强功能，才不容易越做越散。
